伏羲引擎技术架构拆解

// ARCHITECTURE & ENGINEERING · 2025.12.20

摘要： 伏羲引擎 (Fuxi Engine) 是一个典型的 AI Native SaaS 产品。它并未简单地套用 LLM（大语言模型）壳子，而是构建了一个完整的“传统算法 + 生成式 AI”混合架构。本文将从前端交互、后端微服务、核心算法逻辑及支付网关集成四个维度，拆解其技术实现。

1. 系统架构全景 (System Overview)

项目采用经典的 B/S 架构，前后端分离设计。核心通过 RESTful API 进行数据交互，针对 AI 生成部分采用了 Server-Sent Events (SSE) 风格的流式传输技术，以确保在大模型推理时的秒级响应体验。

  • Frontend: Native HTML5/CSS3 + Vanilla ES6 JavaScript (追求极致轻量化与加载速度，无重型框架依赖)
  • Backend: Python Flask + Gevent (高并发协程支持)
  • Database: SQLite / SQLAlchemy (轻量级关系型数据存储，索引优化)
  • AI Core: DeepSeek V3 (Via OpenAI Compatible API)
  • Payments: Hybrid Gateway (Lemon Squeezy + Cryptomus Web3 + PayPal)

2. 核心模块拆解

2.1 命理数据数字化 (Digitization)

这是系统的基石，也是区别于普通“算命 GPT”的核心壁垒。系统并非直接让 AI 瞎编，而是先进行严谨的数学推演。

在后端 app.py 中，我重构了完整的神煞字典库（包含天乙贵人、羊刃、桃花、孤辰寡宿等 50+ 种神煞逻辑）。通过多重映射表（Map-Reduce思想），实现了基于“年/月/日/时”四柱的交叉索引。

自主研发了 BaziLogic 类，通过加权打分法（Weighted Scoring System）计算命主能量，将模糊的传统概念转化为 0.0-1.0 的浮点数。

2.2 AI 提示词工程 (Prompt Engineering)

系统采用了 RAG (Retrieval-Augmented Generation) 的轻量化变体。在调用 LLM 之前，系统会将算法算出的“硬数据”（八字格局、五行强弱、核心神煞）动态注入到 System Prompt 中。

2.3 流式响应 (Streaming Response)

为了解决 LLM 推理延迟带来的用户焦虑，前端放弃了传统的 await fetch() 等待模式。

  • 后端实现：利用 Flask 的 stream_with_context 生成器，结合 DeepSeek 的 stream=True 参数，将 Token 实时推送到前端。
  • 前端实现：使用 TextDecoder 处理二进制流，并实现了一个打字机效果队列（Typewriter Queue），平滑处理网络抖动。

3. 全球支付网关集成

本项目实现了一个复杂的聚合支付路由，旨在最大化全球转化率与抗审查能力。

3.1 混合支付策略

Fiat (法币)：集成 Lemon Squeezy 和 PayPal。利用 Lemon Squeezy 的 Hosted Checkout 简化税务合规；PayPal 作为兜底方案，通过 OAuth2 获取 Access Token 并二次验证订单状态。

Web3 (加密货币)：集成 Cryptomus，支持 USDT (TRC20/ERC20) 支付。实现了基于 MD5 签名的安全校验机制。

3.2 Webhook 安全闭环

所有支付回调（Webhook）均实现了严格的安全校验，防止“伪造充值”攻击。

4. 总结与展望

伏羲引擎展示了如何用现代全栈技术重构传统领域。它不仅是一个 Web App，更是一套完整的 “数据输入 -> 算法清洗 -> AI 增强 -> 商业变现” 的闭环系统。

未来计划引入 Vector Database (向量数据库)，通过存储命理古籍（如《三命通会》、《滴天髓》）的 Embeddings，进一步提升 AI 解读的专业深度。
