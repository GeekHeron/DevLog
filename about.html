04 / Featured Full-Stack Workflows

跨境电商智能调度 | Amazon Ads & Competitor Defense

广告 & 竞品价格双线防御中枢

覆盖 10 款核心 SKU · 8 项广告指标全自动日报化 · 竞品价格 ±5% 波动实时预警

• 核心痛点： 运营每天要在广告后台、库存表、竞品页面之间手动核对数据，广告花费与库存脱节导致超卖或断货式投放，价格战一旦错过响应窗口就是实打实的利润损失。
• 全栈方案： 广告日报脚本同时拉取"运营数据"与"库存追踪"两张飞书多维表，按 SKU 自动匹配在库数量——优先取 FBA 在库，为 0 时自动切换万邑通仓，并支持同前缀子 SKU（如颜色变体）库存自动聚合，管理层不用再手动加总。竞品价格侧则通过紫鸟指纹浏览器 + Playwright 接管真实浏览器环境抓取 Amazon 前台价格，与前一日历史价格比对，波动 ≥±5% 自动标红并推送飞书卡片，无需人工盯盘。

企业级库存与物流风控 | Inventory & Logistics Risk Control

全链路库存与物流风控体系

延期订单自动锁定 TOP 5 高危单 · 超期库龄按 3/6 个月双阈值分层预警

• 核心痛点： 长周期压货和物流延期往往要等到月结才被发现，仓储、物流、运营三方数据分散在不同表格里，异常单据容易被淹没在大量正常数据中。
• 全栈方案： 物流延期脚本按"计划到仓时间"与当前日期自动计算延期天数，剔除已到仓订单后按延期天数降序取 TOP 5 推送并 @ 责任人，管理层第一眼就能看到最紧急的单子，而不是被几十条数据淹没。库龄侧同时监控 3 个月与 6 个月两张超期表，配合产品货值周报形成"哪些货压了多久、值多少钱"的完整视图，全部按周/日自动推送，不需要人工去查表汇总。

工程韧性案例 | When the Official API Breaks

官方 API 失效后：RPA 无缝补位的差评监控

ERP 官方网关认证失效当天完成 RPA 方案切换，监控未中断一天

• 核心痛点： 差评监控原本依赖积加 ERP 官方 API，但对方网关出现认证失效、长期不可用的情况——这在依赖第三方接口的自动化系统里是常见的"断供"风险，很多方案在这一步就直接停摆。
• 全栈方案： 不等官方修复，直接把数据抓取层换成 Playwright RPA，模拟真实登录态从 ERP 前台抓取差评数据，上层的飞书卡片生成与推送逻辑完全复用，业务方几乎无感知。这也是全栈工程师和"只会调 API"的本质区别——接口说停就停，但业务监控不能跟着停。

跨部门运营数据中枢 | Multi-Report Automation Matrix

自动化运营报表矩阵：广告 / 销售 / 物流 / 库龄 / 竞品 / 差评

7 类日报/周报/预警统一中枢 · 支持 --dry-run 预演，不触达真实群聊即可验收

• 核心痛点： 团队每天要在飞书多维表、ERP 后台、Amazon 卖家后台之间手动核对广告花费、库存、物流时效、竞品价格等多类数据，任何一类漏看都可能导致断货或资金占用，人工日报耗时且容易滞后。
• 全栈方案： 交付了一整套定时脚本矩阵，统一从飞书多维表（Bitable）拉取结构化数据，用 Playwright + 指纹浏览器补齐无官方 API 的抓取场景，再统一汇总为飞书 Interactive Card 推送到对应责任人群：广告日报、销售日报、空运物流汇总、物流延期预警、超期库龄统计、产品货值周报、竞品价格波动监控、ERP 差评监控。所有脚本支持 --dry-run 模式，可在不触达真实群聊的情况下预演输出，方便迭代和验收。
• 工程细节： 抓取层与推送层解耦（各脚本拆成 scrape_*.py / report_*.py），单个数据源故障不影响其余报表按时推送；还有一个脚本专门用 Playwright 操作飞书多维表连接器的配置页面（这块飞书没开放官方 API），自动把"查询时间"参数改成前一天并保存，把原本必须人工点击的操作也纳入无人值守流程。

05 / Workflow at a Glance

以上案例背后是同一套可复用的闭环方法论——不管业务场景是广告竞价还是库存清算，交付的系统都遵循相同的四段式结构：

采集层保证数据不失真，决策层承担可解释的规则/模型判断，执行层直接写回企业已有系统而非另建孤岛，回环层把最终裁量权留给业务负责人——这也是"全栈交付"和"单点接 API"的本质区别。
