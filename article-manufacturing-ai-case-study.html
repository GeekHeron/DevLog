企业级 AI Agent 平台落地案例：从数据治理到业务闭环

SEO 关键词：制造业 AI 案例, 企业 AI Agent 平台, 数据治理, RAG, LangGraph, Ontology, 制造业数字化转型
DATE: 2026.07.18 AUTHOR: GEEKHERON CATEGORY: CASE STUDY

我和一家注塑件制造工厂有过一次深度交流。他们的需求听起来很简单："上 AI，接个 DeepSeek，套个 LangChain Agent，完事。"

但调研了三周之后，我发现根本不是那回事。这家工厂的数据源多到吓人：

  • MES（生产制造执行）—— 工单、工序、设备、良率
  • ERP（企业资源计划）—— 采购单、库存、财务
  • CRM（客户关系）—— 客户信息、合同、需求
  • 文件服务器—— 工艺文件、SOP、质量标准、设备图纸
  • 钉钉—— 审批流程、项目文档、沟通记录

最大问题？不是没有数据，是数据之间互相不认识。

MES 说的"产品编码 100-A"，到 ERP 叫"物料编码 M001"，到 CRM 叫"SKU A100"。AI 看到三个字段，以为有三个东西——其实是一个。

所以结论很清晰：不要一上来就想"做一个万能 Agent"，正确的路径应该是：

数据治理 → 企业知识层 → RAG → 工具封装 → Agent 编排 → 业务闭环

这篇文章就是这个案例的完整拆解。它既是我做交付的真实路径，也是一个可以写进简历的完整项目叙事。

一、先做数据接入层

大多数制造企业说"做 AI"，第一反应是训练模型或接 API。但我选择从最底层开始——先把数据打通。

结构化数据接入

MES、ERP、CRM 这类业务系统的数据是结构化的，有清晰的表和字段。接入方案有三个：

  • API 直连——如果系统有 RESTful 接口
  • 数据库只读从库——拉一份只读副本，不影响生产
  • ETL 管道——定时批处理，清洗和标准化

以 MES 的生产表为例：

┌─────────────┬──────────────┬──────┬──────┬──────┬──────┐ │ 订单号 │ 产品 │ 工序 │ 设备 │ 良率 │ 时间 │ ├─────────────┼──────────────┼──────┼──────┼──────┼──────┤ │ MO-20260701 │ 注塑件-A100 │ 工序3 │ 机台5 │ 95% │ 14:30│ └─────────────┴──────────────┴──────┴──────┴──────┴──────┘

统一接入到 PostgreSQL（事务查询）和 ClickHouse（分析查询），构建企业级数据仓库。这一步做完，就解决了"数据散落"的问题。

非结构化数据接入

PDF 工艺文件、Word 标准作业流程、Excel 质量记录、设备图纸——这些数据无法直接用 SQL 查，需要另一条路线：

📄 PDF / Word / Excel / 图纸
▼
🔍 解析 + OCR + 表格提取
▼
✂️ Chunking 切分
▼
🧬 Embedding 向量化
▼
🗄 向量数据库（Milvus / pgvector）

数据接入层做完之后，数据不再散落在各系统里了，但问题还没完全解决：AI 还是不知道怎么用这些数据。

二、建立企业知识层：区分结构化与非结构化

数据接进来了，但不要把所有东西直接丢给 Agent。需要分类处理：

结构化知识 → 数据库查询

这类知识有明确的字段定义和关系结构，适合 SQL 直接查。比如：

"订单 A 的当前生产状态是什么？" -- 一个 JOIN 就够了 SELECT so.order_status, p.completion_rate, p.defect_rate FROM sales_orders so JOIN production_orders p ON so.order_id = p.order_id WHERE so.order_id = 'MO-20260701';

不需要向量检索，不需要大模型推理。数据库就是最好的答案。

非结构化知识 → RAG 检索

《注塑工艺规范》、《质量异常处理流程》这类文档，适合通过 RAG 检索：

  • LlamaIndex 做文档管理和索引
  • Milvus / pgvector 做向量存储
  • LangChain Retriever 做检索接入

推荐技术栈：先上 pgvector（和 PostgreSQL 同库，省一个运维），量级上来了再切 Milvus。

三、做业务知识模型（Ontology）

这是整个案例中最有价值的一步，也是最容易被跳过的。

前面说过同一个东西在三个系统里叫三个名字。如果不解决这个语义对齐问题，AI 永远不可能做对多系统推理。

解决方案是：建立企业业务本体（Ontology）。

核心实体

产品 | 客户 | 订单 | 物料 | 设备 | 工序 | 员工 | 供应商

属性定义

产品: { 编号, 型号, 规格, 客户, BOM } 设备: { 编号, 状态, 位置, 维护周期, 上次停机 } 质量记录: { 缺陷类型, 严重等级, 原因, 处理方案, 责任人 }

有这个模型之后发生了什么改变？

没有 Ontology 时：

用户问"为什么最近客户 A 的订单总是延期？"

Agent 调用了一个 Tool，查了订单表，返回"订单状态：延迟"。然后呢？不知道原因。因为 Agent 不知道订单和生产线、设备、物料之间的关系。

有 Ontology 后：

Agent 理解了实体之间的链路：

客户A → 订单B → 产品C → 工单D → 工序3 → 设备E(机台5) → 异常记录F

然后自动拆解任务：

  1. 查订单 B 的当前状态和计划交期
  2. 查生产工单 D 的进度和完成率
  3. 查设备 E（机台 5）的运行记录
  4. 查工序 3 的历史质量异常
  5. 查物料供应是否正常

这就是知识驱动 Agent——不是黑盒推理，而是有路径、可追溯、可解释的推理。

四、把企业能力封装成 Tool

数据和知识层建好之后，才进入 Agent 层。

核心原则：不要让 Agent 直接访问数据库。

每一层的能力都封装为独立的 Tool，像微服务一样暴露给 Agent：

get_production_status(order_id)
查询当前生产订单状态、完成率、异常情况
来源：MES
get_quality_issue(product_id)
查询产品质量异常和历史处理方案
来源：MES + 知识库
get_inventory(material_id)
查询物料库存和供应商交期
来源：ERP
get_customer_history(customer_id)
查询客户历史订单和交付记录
来源：CRM
query_process_sop(keyword)
检索工艺规范 / SOP 文件
来源：RAG 知识库

Agent 看到的世界就是这几个 Tool 组成的：生产查询、质量分析、库存查询、客户分析、工艺知识检索。

至于背后是 MES 数据库直连还是 ERP API 调用，Agent 完全不关心。这就是标准的适配器模式。

五、LangChain + LangGraph 编排 Agent

到了这一步，LangChain 才登场——作为 Agent 编排层，负责把 Tool 串起来。

简单场景：LangChain Agent

用户问"订单 A 为什么延期？"

👤 用户：订单A为什么延期？
▼
LangChain Agent (ReAct)
▼
1️⃣ get_production_status('MO-A') → 工序3异常
2️⃣ get_quality_issue('注塑件-A100') → 历史异常记录
3️⃣ query_process_sop('工序3 异常处理') → SOP方案
▼
📋 生成：延期原因 + 解决方案 + 恢复步骤

Agent 按步骤调用 Tool，将每一步的返回组合成完整答案。这就是 ReAct 模式——思考 → 行动 → 观察 → 推理。

复杂场景：LangGraph 流程管理

但在制造业，很多场景不是简单的问答，而是有状态的工作流。

比如"订单异常分析"这个场景，天然就是一个流程：

🚀 开始：收到异常警报
▼
📋 查询订单信息
▼
❓ 是否延期？
▼
⚙️ 查询生产进度 + 质量数据
▼
❓ 是否有已知处理方案？
是 → 套用历史方案
否 → 触发研发介入
▼
✋ 人工审批确认
▼
✅ 结束：生成报告 + 更新知识库

这不是聊天，是流程。有节点、有分支、有状态转换、有人工介入点。LangGraph 的有向图（DAG）模型天然适合这类场景。

所以在实际交付中，我选择：

  • 简单场景（单轮查询、信息聚合）→ LangChain Agent（ReAct）
  • 复杂场景（多步骤、人工审批、状态流转）→ LangGraph

核心思想：每一层只做一件事。

  • 数据层不关心上层怎么用
  • 知识层只管理语义
  • Tool 层只封装能力
  • Agent 层只做编排

七、不要做"大一统 Agent"

制造企业做 AI 最容易犯的错：做一个万能 AI 助手，什么都能问。这想法很诱人，但大概率死在半路上。

我的建议是先做三个垂直 Agent，每个只服务一个角色：

1️⃣ 生产运营 Agent

  • 用户：生产经理
  • 能力：查订单状态、查产线产能、查异常工单、查生产日报

2️⃣ 质量 Agent

  • 用户：质量经理
  • 能力：查不良数据、查缺陷原因、查 SOP 规范、查历史处理方案

3️⃣ 销售交付 Agent

  • 用户：业务经理
  • 能力：查客户信息、查订单进度、查预计交期、查应收账款

每个 Agent 只做一件事。做稳了再合并、交叉、扩展。最终聚合成一个企业智能助手，但那是后面的事。

八、这个项目给你的简历带来什么

如果你按照这个架构去做一个真实项目，你的定位将从"AI 自动化工程师"直接接近"制造业 AI 应用架构师 / AI 解决方案工程师"。

这个场景比纯聊天机器人有价值得多，因为它真正涉及：

  • RAG — 企业非结构化知识检索
  • Agent — 多 Tool 编排与 ReAct
  • Skill/Tool 封装 — 企业系统适配层
  • 企业系统集成 — MES / ERP / CRM / 钉钉
  • 业务知识建模 — Ontology / 本体建模

这些正是现在企业愿意付费的方向。纯聊天机器人在卷价格，但能搞定企业系统集成的人永远稀缺。

九、推荐的落地路线

第一阶段：RAG + Tool

先解决"能查、能问、能执行"。接数据、封装 Tool、搭 RAG 知识库。这个阶段价值最大、风险最小。

第二阶段：业务本体

建立企业核心实体的映射关系。让 AI 理解"100-A = M001 = A100"是同一个东西。

第三阶段：GraphRAG + LangGraph

复杂流程编排和多跳推理。设备停机 → 工序延迟 → 订单延期这条链路，AI 能推出来。

不要一开始就做完整的知识图谱。很多项目死在这里。本体是逐步演化的，不是一次建成的。

制造业天然适合知识图谱——BOM、工艺路线、设备关系、供应链、质量追溯——这些本质就是图。但做得太大、太快，反而会成为做不完的坑。

渐进式推进，每次只解决一个痛点。这是企业 AI 落地最朴素的真理。
